# 2 · Key hierarchy

## 2.1 Overview

The assets sub-protocol uses a two-level key hierarchy:

```
Recipient X25519 key  ─── wraps ──→  AMK  ─── encrypts ──→  asset bytes
   (sphere key                       (per
    or grantee key)                  asset)
```

Each level serves a distinct purpose:

- **Recipient key** is the X25519 public key of an authorized reader.
  In the typical case it is a sphere key listed in the subject's DID
  document; in the delegate case it is the X25519 component of a
  mandate grantee's keypair.
- **AMK** (Asset Master Key) is the per-asset AEAD key that encrypts
  the asset's bytes. It is the only place actual decryption power
  lives; everything else is a wrap of the AMK.

This is **one level shallower** than the data sub-protocol's
`sphere → CMK → DEK` chain. The reason is structural: an asset is
atomic. There is no collection-level grouping to amortize a master key
across, and there is no per-record granularity to address with a
distinct key. The asset's bytes ARE the data; the AMK is its key.
Adding an intermediate "asset master master key" would be ceremonial.

Public assets skip the AMK entirely — they are stored as plaintext
bytes and integrity-verified solely by `sha256_of_plaintext` (§2.6).

## 2.2 Recipient keys — no new sphere key

> **Decision (revisable):** v0.1 does NOT introduce a new `#assets`
> sphere key. Assets inherit the recipient set of their attaching
> context. An asset attached to the Ethos `circle` zone is wrapped for
> `#circle-kex` (and any circle-zone grantees); an asset attached to a
> data collection is wrapped for `#data-kex` (and any collection
> grantees).
>
> **Rationale.** A dedicated `#assets` sphere key would introduce a
> third class of access that would either (a) shadow the existing zone
> and collection access controls — confusing — or (b) duplicate them —
> useless. The attaching context's recipient set is already the natural
> security boundary the asset belongs to.
>
> **Alternative.** Introduce a `#assets` sphere key as a unified gateway
> to all of the subject's assets, independent of zone or collection
> membership. Benefit: one rotation rotates access to all assets.
> Cost: divorces asset access from the context that motivated the
> attachment, requires the user to manage three classes of sphere keys
> instead of two, and complicates the RecipientResolver (§1.5) which
> would have to merge two sources.
>
> **Pending review.** Open to revision if implementation experience
> shows users want a single gateway to all assets. The migration would
> be additive (existing wraps stay; the new `#assets` recipient is
> added).

Practically, the X25519 recipient public keys used by the assets
sub-protocol are:

- `did:aithos:<subject>#public` — for public-zone-attached assets
  (effectively unused because public assets carry no AMK; this is here
  for symmetry only).
- `did:aithos:<subject>#circle-kex` — for circle-attached assets.
- `did:aithos:<subject>#self-kex` — for self-attached assets.
- `did:aithos:<subject>#data-kex` — for data-collection-attached assets.
- `did:key:…<grantee>-kex` or `did:aithos:<agent>#<key>` — for
  delegate grantees (chapter 04).

## 2.3 Asset Master Key (AMK)

### 2.3.1 Construction

An AMK is a 32-byte symmetric key generated by a cryptographically
secure random number generator at upload time. Implementations MUST use
the system CSPRNG.

```ts
const amk: Uint8Array = crypto.getRandomValues(new Uint8Array(32));
```

The AMK is never stored in the clear. It is wrapped for each recipient
the RecipientResolver returns at upload time (§1.5) and only those
wraps are persisted in the asset's metadata document. The AMK itself
is held in client memory only during the upload session and during
subsequent fetches.

### 2.3.2 Asset bytes encryption

The asset bytes are encrypted with XChaCha20-Poly1305 IETF under the
AMK:

```
plaintext_bytes  = the actual asset content (image, PDF, …)
nonce            = random(24 bytes)
aad              = "aithos-asset-v1\0" ‖ utf8(asset_urn)
ciphertext       = XChaCha20-Poly1305.encrypt(
                     key       = amk,
                     nonce     = nonce,
                     aad       = aad,
                     plaintext = plaintext_bytes
                   )
```

The 24-byte nonce is **prepended** to the ciphertext before the bytes
are uploaded to S3. The on-disk layout of an encrypted asset is:

```
[ nonce: 24 bytes | ciphertext: N bytes | poly1305 tag: 16 bytes ]
```

This is the canonical "nonce-prefix" layout for XChaCha20-Poly1305 in
the Aithos protocol family. A fetcher reads the first 24 bytes as the
nonce, the remainder as the ciphertext+tag, and recovers the plaintext
in one AEAD call after unwrapping the AMK.

The AAD binds the ciphertext to the asset URN. Replaying a ciphertext
under a different `asset_id` (e.g. by an attacker who corrupts the
DynamoDB metadata to point a new asset_id at an old ciphertext) fails
to decrypt: the AEAD authentication tag mismatch under the new AAD
aborts the operation.

### 2.3.3 Wrap construction

The AMK is wrapped for each recipient using the same
X25519-HKDF-SHA256-AEAD construction used elsewhere in Aithos (Ethos
spec §3.6, data spec §2.3.2). For a given recipient identified by an
X25519 public key `recipient_pk`:

```
ephemeral_sk, ephemeral_pk = X25519.keypair()                  // fresh
shared_secret              = X25519(ephemeral_sk, recipient_pk)
salt                       = "aithos-assets-amk-wrap-v1"
info                       = recipient_did_url
wrap_key                   = HKDF-SHA256(shared_secret, salt, info, 32)
wrap_nonce                 = random(24 bytes)
aad                        = "aithos-assets-amk-v1\0" ‖ utf8(asset_urn) ‖ "\0" ‖
                             utf8(recipient_did_url)
wrapped_key                = XChaCha20Poly1305.encrypt(
                               key       = wrap_key,
                               nonce     = wrap_nonce,
                               aad       = aad,
                               plaintext = amk
                             )
```

The wrap is serialized as:

```json
{
  "recipient": "did:aithos:z6Mkr…#self-kex",
  "alg": "x25519-hkdf-sha256-aead",
  "ephemeral_public": "z6LS…",
  "wrap_nonce": "bK3x…",
  "wrapped_key": "cP2r…"
}
```

| Field | Type | Description |
|---|---|---|
| `recipient` | string | DID URL of the recipient's X25519 verification method. |
| `alg` | string | MUST be `"x25519-hkdf-sha256-aead"` in v0.1. |
| `ephemeral_public` | string (multibase) | The ephemeral X25519 public key generated for this wrap. |
| `wrap_nonce` | string (base64) | The 24-byte XChaCha20-Poly1305 nonce. |
| `wrapped_key` | string (base64) | The AEAD ciphertext of the AMK (32 bytes + 16-byte tag). |

The AAD binds the wrap to both the asset URN and the recipient DID URL,
preventing a wrap from being replayed across assets or recipients.

### 2.3.4 Wrap list

An asset's `amk_envelope` (§1.2.2) contains the array of wraps:

```json
{
  "alg": "xchacha20poly1305-ietf",
  "nonce": "<nonce of the ciphertext on S3>",
  "wraps": [
    {
      "recipient": "did:aithos:z6Mkr…#self-kex",
      "alg": "x25519-hkdf-sha256-aead",
      "ephemeral_public": "z6LS…",
      "wrap_nonce": "…",
      "wrapped_key": "…"
    },
    {
      "recipient": "did:key:z6Mk…recruiter-acme-kex",
      "alg": "x25519-hkdf-sha256-aead",
      "ephemeral_public": "z6LT…",
      "wrap_nonce": "…",
      "wrapped_key": "…"
    }
  ]
}
```

Note that the `nonce` field at the envelope level is the **bytes
encryption nonce** (used in §2.3.2) — not a wrap nonce. Each wrap
carries its own independent `wrap_nonce`. The envelope's `nonce` is
not strictly required (it could be read from the first 24 bytes of the
S3 object), but it is repeated in the metadata document so a fetcher
can size the byte range request precisely before decrypting.

Invariants the platform MUST enforce:

- **At least one wrap.** A private asset MUST have at least one wrap
  for one of the subject's sphere keys. An asset whose only wraps are
  for revoked grantees is undefined behaviour; the platform MUST
  refuse to publish such a state.
- **No duplicate recipients.** Each `recipient` DID URL appears at most
  once in the wraps array.
- **Recipient resolution.** Every `recipient` MUST be a verification
  method that resolves at the time of the wrap operation: either a
  sphere key listed in the subject's DID document, or a grantee key
  declared in an active mandate the subject has issued.

### 2.3.5 Unwrap procedure

A holder of the X25519 private key corresponding to one of the wraps
unwraps the AMK as follows:

```
match wrap in asset.amk_envelope.wraps where recipient corresponds to my key
if not found: ABORT  (recipient is not authorized on this asset)

shared_secret = X25519(my_x25519_sk, wrap.ephemeral_public)
salt          = "aithos-assets-amk-wrap-v1"
info          = wrap.recipient
wrap_key      = HKDF-SHA256(shared_secret, salt, info, 32)
aad           = "aithos-assets-amk-v1\0" ‖ utf8(asset_urn) ‖ "\0" ‖
                utf8(wrap.recipient)
amk           = XChaCha20Poly1305.decrypt(
                  key        = wrap_key,
                  nonce      = wrap.wrap_nonce,
                  aad        = aad,
                  ciphertext = wrap.wrapped_key
                )
if decrypt fails: ABORT  (corruption, tampering, or wrong key)
return amk
```

A successful unwrap yields the 32-byte AMK in cleartext, which the
client then uses to decrypt the asset bytes (§2.3.6).

### 2.3.6 Bytes decryption

With the AMK in hand:

```
fetched_blob   = the bytes downloaded from the presigned S3 URL
nonce          = fetched_blob[0..24]
ciphertext     = fetched_blob[24..]
aad            = "aithos-asset-v1\0" ‖ utf8(asset_urn)
plaintext      = XChaCha20-Poly1305.decrypt(
                   key        = amk,
                   nonce      = nonce,
                   aad        = aad,
                   ciphertext = ciphertext
                 )
if decrypt fails: ABORT  (corruption or tampering)
verify sha256(plaintext) == asset.sha256_of_plaintext
if mismatch: ABORT  (content does not match the claimed hash)
return plaintext
```

The post-decrypt SHA-256 check is mandatory. It defends against a
scenario in which the AEAD passes (e.g. because the AAD and nonce are
consistent) but the ciphertext was substituted by another valid
ciphertext from the same asset's history — a hypothetical scenario
under AMK rotation. Verifying against the declared
`sha256_of_plaintext` closes that gap.

### 2.3.7 Authorize a new recipient

To authorize a new grantee on an asset:

```
1. Unwrap the AMK using one of the owner's own wraps (§2.3.5).
2. Compute a new wrap of the AMK for the new recipient's X25519 public
   key (§2.3.3).
3. Append the new wrap to the asset's amk_envelope.wraps array.
4. Persist the updated metadata via `aithos.assets.authorize_grantee`.
```

Cost: one X25519 keypair generation + one ECDH + one HKDF + one AEAD
encryption. Constant in asset size. The S3 object is not touched: its
bytes stay byte-identical.

Server-side, the platform verifies:

- The caller's envelope signature is valid.
- The caller is the asset's owner.
- The new recipient's DID URL is well-formed.
- No existing wrap targets the same recipient (idempotency).

A gamma entry `assets.authorize_grantee` is emitted (chapter 08).

### 2.3.8 Revoke a recipient

To revoke a grantee:

```
1. Remove the recipient's wrap from the asset's amk_envelope.wraps.
2. Persist via `aithos.assets.revoke_grantee`.
3. OPTIONALLY: rotate the AMK (§2.5.2) if forward secrecy is required.
```

Step 3 is the policy decision. Without rotation, the revoked grantee
can no longer fetch the asset via the PDS (the PDS enforces the removed
wrap by refusing to issue presigned URLs to envelopes signed under the
revoked mandate), but if the grantee retained a copy of the AMK from
prior sessions, it can still decrypt any ciphertext blob it has cached.
With rotation, the new AMK is unknown to the revoked grantee and any
new ciphertext is opaque.

> **Decision (revisable):** v0.1 specifies AMK rotation at revoke time
> as OPTIONAL by default, declarable per-asset at upload time via a
> `forward_secrecy: "strict" | "best_effort"` flag (defaults to
> `"best_effort"`).
>
> **Rationale.** AMK rotation on revoke costs O(asset size) — the
> entire ciphertext is re-encrypted under the new AMK. For routine
> mandate cycles, this is wasteful. For high-stakes assets
> (signed contracts, IDs), the owner may opt into `"strict"` at upload
> time and pay the cost where it matters.
>
> **Pending review.** The default flag value, the rotation procedure
> details, and the boundary between `best_effort` and `strict` are open
> to revision based on real-world usage.

## 2.4 Public assets

Public assets carry no AMK and no wraps. The asset metadata document
has `encrypted: false` and lacks the `amk_envelope` field.

The S3 object holds the plaintext bytes directly:

```
[ plaintext_bytes: N bytes ]
```

No prefix nonce, no AEAD framing.

Integrity is verified by computing `sha256(fetched_bytes)` and comparing
to the metadata's `sha256_of_plaintext`. The platform MUST refuse a
`complete_upload` whose computed hash diverges (§1.4.2).

The Ethos manifest's signature transitively vouches for the
`sha256_of_plaintext` of every asset referenced from a section in that
manifest, so a counterparty consuming a public Ethos bundle can verify
asset integrity end-to-end without trusting the asset PDS.

## 2.5 Lifecycle and rotation

### 2.5.1 AMK reuse across reads

The AMK is generated once at upload and persisted in wrapped form. Every
authorized reader unwraps a copy on demand and uses it to decrypt the
bytes. There is no per-fetch AMK rotation; the AAD-bound nonce in the
ciphertext is sufficient to bind each ciphertext to its `asset_urn`.

### 2.5.2 AMK rotation

AMK rotation is an owner-initiated operation invoked via
`aithos.assets.rotate_amk` (chapter 05 §5.4.10). The procedure:

1. Generate a new AMK `amk_new`.
2. Re-encrypt the asset bytes under `amk_new` with a fresh nonce (the
   plaintext is recovered from the old ciphertext using `amk_old`,
   which the owner unwraps from one of their wraps).
3. For every recipient currently in the asset's wrap list (except those
   being explicitly revoked in the same operation): wrap `amk_new` for
   that recipient.
4. Atomically replace:
   - The S3 object's bytes with the new ciphertext.
   - The asset metadata's `amk_envelope` with the new envelope.
   - The asset metadata's `nonce` with the new nonce.
5. Emit a gamma entry `assets.amk_rotated` carrying the new wrap list
   hash and a reference to the old wrap list (for audit).

Rotation cost is O(asset size) for the byte re-encryption + O(N) on
recipients for the wraps. The operation requires the owner to have the
plaintext available locally — which they do, since they hold a wrap of
the old AMK and can recover the plaintext.

The platform MUST treat the rotation as atomic from a reader's
perspective: a `get_asset` call sees either the old envelope + old
bytes consistently, or the new envelope + new bytes consistently, never
a mix. The reference S3+DynamoDB implementation achieves this via S3
conditional writes (preserving the previous version under versioning)
and a DynamoDB transaction that updates the metadata only after the new
S3 object has been written.

### 2.5.3 Sphere key rotation

Rotation of the subject's underlying sphere key (`#circle`, `#self`,
`#data`) follows the Ethos sub-protocol's standard sphere rotation flow
(Ethos spec §1.6). For the assets sub-protocol:

1. The DID document update advertises a new sphere verification method.
2. For every asset whose `amk_envelope.wraps[]` contains a wrap for
   the old sphere key, the owner MUST issue
   `aithos.assets.rotate_owner_wrap` (chapter 05 §5.4.11), which
   re-wraps the AMK for the new sphere key. This is a single wrap
   operation per asset, not a re-encryption.
3. Recipient wraps for application grantees are unaffected; mandates
   bind to grantee keys, not to the subject's sphere keys.

Until the owner wrap is rotated, the subject loses access to that
specific asset (cannot unwrap the AMK with the new sphere key).
Recovery is possible if the subject has another wrap (a recovery
keystore, a delegate with full-access scope). Implementations SHOULD
warn the user before a sphere rotation if any asset lacks a non-rotated
wrap.

For a subject with thousands of assets, this is O(N). The reference
implementation MAY expose a batch primitive
`aithos.assets.bulk_rotate_owner_wrap` for this scenario; it is
specified informatively in chapter 05 §5.4.11.1.

## 2.6 Integrity for public assets

A public asset's integrity is established by its
`sha256_of_plaintext`. Specifically:

- At upload, the client computes the hash, declares it in
  `init_upload`, and the platform verifies it in `complete_upload`
  against the actually-uploaded bytes (§1.4.2).
- At reference time, the Ethos manifest's `manifest.assets[i].sha256_of_plaintext`
  is signed by the manifest signature, transitively vouching for the
  asset's content.
- At fetch time, the client recomputes the hash and compares to the
  expected value (which it obtained either from the asset metadata
  document or from the consuming context — typically the Ethos
  manifest).

A platform MAY additionally serve a separate "integrity sidecar" file
on S3 — a small JSON document containing the SHA-256 and a signature by
the subject's `#public` sphere key — for environments where the
consumer cannot reach the asset PDS RPC but can fetch the CloudFront
URL. This is informative; v0.1 does not normatively require it.

## 2.7 Algorithm and parameter table

| Use | Algorithm | Output length | Notes |
|---|---|---|---|
| Random key/nonce generation | OS CSPRNG | varies | Web Crypto, libsodium, OS-provided |
| Key agreement | X25519 (RFC 7748) | 32-byte shared secret | curve25519 ECDH |
| KDF for wrap key derivation | HKDF-SHA256 (RFC 5869) | 32 bytes | salt `"aithos-assets-amk-wrap-v1"`, info = recipient DID URL |
| AEAD for AMK wraps | XChaCha20-Poly1305 IETF | ciphertext + 16-byte tag | 24-byte nonce |
| AEAD for asset bytes | XChaCha20-Poly1305 IETF | ciphertext + 16-byte tag | 24-byte nonce, prepended to ciphertext on S3 |
| Content hash | SHA-256 (RFC 6234) | 32 bytes | computed over plaintext, lower-case hex |
| Signing (envelopes, gamma) | Ed25519 (RFC 8032) | 64-byte signature | reused from Ethos |

A future minor version MAY add post-quantum algorithms; algorithm
identifiers (`alg` field) and the registry allow this without breaking
the wire format.

## 2.8 Test vectors (informative)

Conformant implementations SHOULD publish test vectors covering:

- AMK generation, AMK encryption of known plaintext, wrap for one
  recipient, unwrap with the matching X25519 private key, decryption,
  hash verification, round-trip equality with original.
- AAD binding to a specific `asset_urn`: a ciphertext whose AAD is
  computed with the wrong URN fails to decrypt.
- Wrap AAD binding to a specific recipient: a wrap produced for
  recipient A cannot be decrypted by recipient B even if B has A's
  X25519 private key (because the AAD includes the recipient DID URL).
- Authorize-new-recipient flow: unwrap AMK, wrap AMK for new recipient,
  verify new recipient can unwrap.
- Revoke-recipient flow: removed wrap, attempts to unwrap by the
  revoked party fail with AEAD authentication failure.
- AMK rotation flow: new AMK, new ciphertext, old recipients can still
  unwrap if re-wrapped, revoked recipients cannot.
- Public asset hash verification: a fetched plaintext whose SHA-256
  differs from the claimed hash is rejected.

Test vectors live in `packages/assets-crypto/test/vectors/` once the
reference implementation lands. See [chapter 09 — Threat model](./09-threat-model.md)
for the attack scenarios these vectors are meant to defeat.
