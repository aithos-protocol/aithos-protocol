# 2 · Key hierarchy

## 2.1 Overview

The data sub-protocol uses a three-level key hierarchy:

```
Owner sphere key  ─── wraps ──→  CMK  ─── wraps ──→  DEK  ─── encrypts ──→  payload
   (long-lived,                  (per                (per
    identity-stable)              collection)        record)
```

Each level serves a distinct purpose:

- **Sphere key** is the root of trust. It belongs to the subject's DID
  and is the same key material used throughout the Ethos sub-protocol.
- **CMK** (Collection Master Key) intermediates between the sphere key
  and the records' DEKs. It is the layer that delivers O(1) authorization
  — one wrap of the CMK per application grants access to every record
  in the collection.
- **DEK** (Data Encryption Key) is the actual AEAD key used to encrypt
  one record's payload. There is one DEK per record, and each DEK is
  wrapped only for the CMK.

This chapter specifies the construction, format, and lifecycle of the
CMK and DEK, and the wrap operation that binds keys to recipients.

## 2.2 Sphere key — recap

A subject's DID document declares one or more sphere keys (Ethos spec
§1.2). The data sub-protocol uses a dedicated sphere key:

```
did:aithos:z6Mkr…#data
```

> **Decision (revisable):** v0.1 introduces a new `#data` sphere key
> rather than reusing the existing `#circle` or `#self` keys.
>
> **Rationale.** The `#data` sphere can be rotated independently of the
> Ethos spheres. If a subject's data-access devices (mobile, browser)
> are compromised separately from their Ethos identity device, only the
> `#data` key needs rotation. Conversely, reusing `#circle` would couple
> two access classes — losing access to circle would also lose data
> access, even though the threat models differ.
>
> **Alternative.** Reuse `#circle` for backward compatibility with
> existing Aithos subjects. Cost: harder to scope key rotation. Benefit:
> no DID document modification needed for existing subjects.
>
> **Pending review.** This decision is open to revision based on
> implementation feedback.

The `#data` sphere key is an Ed25519 keypair (for signing) plus an
X25519 keypair (for key agreement), bound together in the DID document
under the verification method id `did:aithos:<subject>#data`. The
X25519 component is exposed at the DID URL `did:aithos:<subject>#data-kex`
for key-agreement operations.

When a subject does not yet have a `#data` sphere key, the first call to
`aithos.data.create_collection` (chapter 05) triggers a DID document
update to add one. The DID document update is signed by the subject's
root key and follows the standard rotation flow (Ethos spec §1.6).

## 2.3 Collection Master Key (CMK)

### 2.3.1 Construction

A CMK is a 32-byte symmetric key generated by a cryptographically secure
random number generator at collection creation time. Implementations MUST
use the system CSPRNG (Web Crypto `crypto.getRandomValues`,
Node `crypto.randomBytes`, equivalent on other runtimes).

```ts
const cmk: Uint8Array = crypto.getRandomValues(new Uint8Array(32));
```

The CMK is never stored in the clear. It is wrapped for each recipient
authorized on the collection and only those wraps are persisted.

### 2.3.2 Wrap construction

A CMK wrap is produced by the X25519-HKDF-SHA256-AEAD construction
already used in the Ethos sub-protocol (Ethos spec §3.6). For a given
recipient identified by an X25519 public key `recipient_pk`:

```
ephemeral_sk, ephemeral_pk = X25519.keypair()                  // fresh
shared_secret              = X25519(ephemeral_sk, recipient_pk)
salt                       = "aithos-data-cmk-wrap-v1"
info                       = recipient_did_url
wrap_key                   = HKDF-SHA256(shared_secret, salt, info, 32)
wrap_nonce                 = random(24 bytes)
aad                        = "aithos-data-cmk-v1\0" ‖ utf8(collection_urn) ‖ "\0" ‖
                             utf8(recipient_did_url)
wrapped_key                = XChaCha20Poly1305.encrypt(
                               key       = wrap_key,
                               nonce     = wrap_nonce,
                               aad       = aad,
                               plaintext = cmk
                             )
```

The wrap is serialized as:

```json
{
  "recipient": "did:aithos:z6Mkr…#data-kex",
  "alg": "x25519-hkdf-sha256-aead",
  "ephemeral_public": "z6LS…",
  "wrap_nonce": "bK3x…",
  "wrapped_key": "cP2r…"
}
```

| Field | Type | Description |
|---|---|---|
| `recipient` | string | DID URL of the recipient's X25519 verification method. |
| `alg` | string | MUST be `"x25519-hkdf-sha256-aead"` in v0.1. |
| `ephemeral_public` | string (multibase) | The ephemeral X25519 public key generated for this wrap. |
| `wrap_nonce` | string (base64) | The 24-byte XChaCha20-Poly1305 nonce. |
| `wrapped_key` | string (base64) | The AEAD ciphertext of the CMK (32 bytes + 16-byte tag). |

The wrap nonce is unique per wrap operation, drawn from a CSPRNG. The
AAD binds the wrap to both the collection URN and the recipient DID URL,
preventing a wrap from being replayed across collections or across
recipients.

### 2.3.3 Wrap list

A collection's CMK metadata (§1.2.2) contains an array of wraps:

```json
{
  "alg": "xchacha20poly1305-ietf",
  "wraps": [
    {
      "recipient": "did:aithos:z6Mkr…#data-kex",
      "alg": "x25519-hkdf-sha256-aead",
      "ephemeral_public": "z6LS…",
      "wrap_nonce": "…",
      "wrapped_key": "…"
    },
    {
      "recipient": "did:key:z6Mk…switchia-grantee-kex",
      "alg": "x25519-hkdf-sha256-aead",
      "ephemeral_public": "z6LT…",
      "wrap_nonce": "…",
      "wrapped_key": "…"
    },
    {
      "recipient": "did:key:z6Mk…email-app-grantee-kex",
      "alg": "x25519-hkdf-sha256-aead",
      "ephemeral_public": "z6LU…",
      "wrap_nonce": "…",
      "wrapped_key": "…"
    }
  ]
}
```

Invariants the platform MUST enforce:

- **At least one wrap.** A collection MUST have at least one wrap for
  one of the subject's sphere keys. A collection whose only wraps are
  for revoked grantees is undefined behaviour; the platform MUST refuse
  to publish such a state.
- **No duplicate recipients.** Each `recipient` DID URL appears at most
  once in the wraps array.
- **Recipient resolution.** Every `recipient` MUST be a verification
  method that resolves at the time of the wrap operation: either a
  sphere key listed in the subject's DID document, or a grantee key
  declared in an active mandate the subject has issued.

### 2.3.4 Unwrap procedure

A holder of the X25519 private key corresponding to one of the wraps
unwraps the CMK as follows:

```
match wrap in collection.cmk.wraps where recipient corresponds to my key
if not found: ABORT  (recipient is not authorized on this collection)

shared_secret = X25519(my_x25519_sk, wrap.ephemeral_public)
salt          = "aithos-data-cmk-wrap-v1"
info          = wrap.recipient
wrap_key      = HKDF-SHA256(shared_secret, salt, info, 32)
aad           = "aithos-data-cmk-v1\0" ‖ utf8(collection_urn) ‖ "\0" ‖
                utf8(wrap.recipient)
cmk           = XChaCha20Poly1305.decrypt(
                  key        = wrap_key,
                  nonce      = wrap.wrap_nonce,
                  aad        = aad,
                  ciphertext = wrap.wrapped_key
                )
if decrypt fails: ABORT  (corruption, tampering, or wrong key)
return cmk
```

A successful unwrap yields the 32-byte CMK in cleartext, which the
client then uses to unwrap individual record DEKs (§2.4.3).

### 2.3.5 Authorize a new recipient

To authorize a new application on a collection, an authorized client
(typically the owner) executes:

```
1. Unwrap the CMK using one of its own wraps (§2.3.4).
2. Compute a new wrap of the CMK for the new recipient's X25519 public
   key (§2.3.2).
3. Append the new wrap to the collection's wraps array.
4. Persist the updated collection metadata via `aithos.data.authorize_app`.
```

Cost: one X25519 keypair generation + one ECDH + one HKDF + one AEAD
encryption. Constant in collection size. The collection's records are
not touched: their ciphertexts and `dek_wrapped_for_cmk` fields stay
byte-identical.

Server-side, the platform verifies:

- The caller's envelope signature is valid.
- The caller is the collection's owner (envelope `iss == subject_did`
  and signing key is a sphere key listed in the subject's DID document).
- The new recipient's `recipient` DID URL is well-formed.
- No existing wrap targets the same `recipient` (idempotency).

A gamma entry `data.collection.authorize_grantee` is emitted (chapter 08).

### 2.3.6 Revoke a recipient

To revoke an application:

```
1. Remove the recipient's wrap from the collection's wraps array.
2. Persist via `aithos.data.revoke_app`.
3. OPTIONALLY: rotate the CMK (§2.5.2) if forward secrecy is required.
```

Step 3 is the policy decision. Without rotation, the revoked grantee can
no longer fetch new collection state via the PDS (the PDS enforces the
removed wrap by refusing service to envelopes signed under the revoked
mandate), but if the grantee retained a copy of the CMK from prior
sessions, it can still decrypt any record ciphertext it has cached.
With rotation, the new CMK does not unwrap the old DEKs unless those
DEKs are also rotated and re-wrapped — which means re-encrypting the
records.

> **Decision (revisable):** v0.1 specifies CMK rotation at revoke time
> as OPTIONAL by default, declarable per-collection at create time via a
> `forward_secrecy: "strict" | "best_effort"` flag.
>
> **Rationale.** The cost of CMK rotation is O(N) on records (each DEK
> must be re-wrapped under the new CMK; each ciphertext must be
> re-encrypted to deny the revoked grantee access to past values).
> Forcing this cost on every revoke regardless of threat model is
> unjustified for typical business records. Subjects handling regulated
> data (health, finance) can opt into `strict` at collection creation
> and pay the cost where it matters.
>
> **Pending review.** The default flag value, the rotation procedure
> details, and the migration path between modes are open to revision.

## 2.4 Data Encryption Key (DEK)

### 2.4.1 Construction

A DEK is a 32-byte symmetric key generated by the CSPRNG at record
insertion time:

```ts
const dek: Uint8Array = crypto.getRandomValues(new Uint8Array(32));
```

There is one DEK per record. The DEK is never stored in the clear; it is
wrapped only for the collection's current CMK and persisted alongside
the record's ciphertext.

### 2.4.2 Wrap of the DEK under the CMK

The DEK is wrapped using a direct AEAD construction (no key agreement,
since the CMK is already a symmetric secret known to all authorized
parties):

```
wrap_nonce        = random(24 bytes)
aad               = "aithos-data-dek-v1\0" ‖ utf8(subject_did) ‖ "\0" ‖
                    utf8(collection_name) ‖ "\0" ‖ utf8(record_id)
dek_wrapped       = XChaCha20Poly1305.encrypt(
                      key        = cmk,
                      nonce      = wrap_nonce,
                      aad        = aad,
                      plaintext  = dek
                    )
```

The wrap is stored as a single field in the record's payload envelope
(§1.3.4) under the name `dek_wrapped_for_cmk`. The wrap nonce is
prepended to the AEAD ciphertext in serialization:

```
serialized_wrap = base64(wrap_nonce ‖ dek_wrapped)
```

(The 24-byte nonce is positionally identifiable as the first 24 bytes
of the decoded buffer.)

### 2.4.3 Unwrap of the DEK

A client holding the CMK extracts the DEK as follows:

```
buffer        = base64_decode(record.payload.dek_wrapped_for_cmk)
wrap_nonce    = buffer[0..24]
dek_ciphertext = buffer[24..]
aad           = "aithos-data-dek-v1\0" ‖ utf8(subject_did) ‖ "\0" ‖
                utf8(collection_name) ‖ "\0" ‖ utf8(record_id)
dek           = XChaCha20Poly1305.decrypt(
                  key        = cmk,
                  nonce      = wrap_nonce,
                  aad        = aad,
                  ciphertext = dek_ciphertext
                )
if decrypt fails: ABORT
return dek
```

The AAD binding to `(subject_did, collection_name, record_id)` ensures
the wrapped DEK cannot be replayed into a different record.

### 2.4.4 Payload encryption

Once the DEK is in hand (either freshly generated for an insert or
unwrapped for a read), the record's payload is encrypted as follows:

```
plaintext        = JCS-canonicalize(payload_object)
                   // payload_object contains only encrypted fields,
                   // per the schema (chapter 03)
nonce            = random(24 bytes)
aad              = "aithos-data-record-v1\0" ‖ utf8(subject_did) ‖ "\0" ‖
                   utf8(collection_name) ‖ "\0" ‖ utf8(record_id)
ciphertext       = XChaCha20Poly1305.encrypt(
                     key       = dek,
                     nonce     = nonce,
                     aad       = aad,
                     plaintext = plaintext
                   )
```

The payload object is JCS-canonicalized (RFC 8785) before encryption so
that two implementations encrypting the same logical payload produce the
same plaintext bytes — useful for content addressability and integrity
testing.

## 2.5 Lifecycle and rotation

### 2.5.1 DEK rotation on update

When a record is updated, the implementation has two options for the DEK:

1. **Reuse the DEK with a fresh nonce.** Cheaper. Acceptable as long as
   the (DEK, nonce) pair is never reused — XChaCha20-Poly1305's 24-byte
   nonce gives ~2⁹⁶ usable nonces per DEK, vastly more than any record
   will ever need.
2. **Rotate the DEK.** New DEK, new wrap, new ciphertext. Costs one
   extra AEAD operation per update.

> **Decision (revisable):** v0.1 specifies DEK reuse with fresh nonce as
> the DEFAULT, with DEK rotation as RECOMMENDED on schema migration
> (when the set of encrypted fields changes).
>
> **Rationale.** A per-update DEK rotation provides no meaningful
> additional security as long as nonces are unique. The historical
> XChaCha20-Poly1305 paranoia (avoid (key, nonce) reuse) is satisfied
> by fresh nonces alone. Rotating the DEK costs an extra AEAD per
> update on a hot write path.
>
> **Pending review.** Implementations MAY choose to always rotate DEKs
> for defense-in-depth; the spec permits this.

### 2.5.2 CMK rotation

CMK rotation is a deliberate, owner-issued operation that:

1. Generates a new CMK.
2. For every record in the collection: unwraps the existing DEK under
   the old CMK, re-wraps it under the new CMK. The record ciphertext
   itself is **not** re-encrypted unless §2.3.6's `strict` forward
   secrecy mode is in effect.
3. For every authorized recipient (the wrap list): wraps the new CMK
   for that recipient.
4. Replaces the collection's CMK envelope and all record `dek_wrapped_for_cmk`
   fields atomically. Emits a single gamma entry `data.collection.rotate_cmk`
   carrying both the new wrap list and the old CMK hash (for audit).

Rotation cost is O(N) on records. The operation is interruptible: a
partial rotation leaves the collection in a recoverable state, with
records that haven't yet been re-wrapped still readable under the old
CMK. Atomicity is recommended but not strictly required by the protocol.

The platform MAY refuse to begin a CMK rotation if the previous rotation
is still in progress; concurrent rotations are unsafe (a record might be
re-wrapped twice or not at all).

### 2.5.3 Sphere key rotation

Rotation of the subject's `#data` sphere key follows the Ethos
sub-protocol's standard sphere rotation flow (Ethos spec §1.6). The
practical effect on the data sub-protocol:

1. The DID document update advertises a new `#data` verification method.
2. For every collection the subject owns, the wrap addressed to the old
   `#data-kex` becomes unverifiable. The subject MUST issue an
   `aithos.data.rotate_owner_wrap` call (chapter 05) for each
   collection, which re-wraps the CMK for the new sphere key. This is
   a single wrap operation per collection, not per record.
3. Recipient wraps for application grantees are unaffected; mandates
   bind to grantee keys, not to the subject's sphere key.

Until the owner wrap is rotated, the subject loses access to the
collection (cannot unwrap the CMK). Recovery is possible if the subject
has another wrap (e.g. a recovery file with a separate wrap, or a
delegate with full-access scope). Implementations SHOULD warn the user
before a sphere rotation if any collection lacks a non-#data wrap.

## 2.6 Algorithm and parameter table

| Use | Algorithm | Output length | Notes |
|---|---|---|---|
| Random key/nonce generation | OS CSPRNG | varies | Web Crypto, libsodium, OS-provided |
| Key agreement | X25519 (RFC 7748) | 32-byte shared secret | curve25519 ECDH |
| KDF for wrap key derivation | HKDF-SHA256 (RFC 5869) | 32 bytes | salt and info per §2.3.2 |
| AEAD for CMK wraps | XChaCha20-Poly1305 IETF | ciphertext + 16-byte tag | 24-byte nonce |
| AEAD for DEK wraps | XChaCha20-Poly1305 IETF | ciphertext + 16-byte tag | 24-byte nonce |
| AEAD for record payloads | XChaCha20-Poly1305 IETF | ciphertext + 16-byte tag | 24-byte nonce |
| Signing (envelopes, gamma) | Ed25519 (RFC 8032) | 64-byte signature | reused from Ethos |
| Canonicalization (payload) | RFC 8785 JCS | UTF-8 bytes | only for pre-encryption canon |

A future minor version MAY add post-quantum algorithms; algorithm
identifiers (`alg` field) and the registry in chapter 03's schema
metadata allow this without breaking the wire format.

## 2.7 Test vectors (informative)

Conformant implementations SHOULD publish test vectors covering:

- CMK generation, wrap for one recipient, unwrap with the matching X25519
  private key, comparison of recovered CMK to original.
- DEK generation, wrap under a known CMK, AAD binding to a specific
  `(subject_did, collection_name, record_id)`, unwrap and comparison.
- Payload encryption + decryption roundtrip on a known schema instance.
- Authorize-new-recipient flow: unwrap CMK, wrap CMK for new recipient,
  verify new recipient can unwrap.
- Revoke-recipient flow: removed wrap, attempts to unwrap by the revoked
  party fail with AEAD authentication failure.
- CMK rotation flow: re-wrap all DEKs, old CMK no longer unwraps.

Test vectors live in `packages/data-crypto/test/vectors/` once the
reference implementation lands. See [chapter 09 — Threat model](./09-threat-model.md)
for the attack scenarios these vectors are meant to defeat.
